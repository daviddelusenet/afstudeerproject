1.3Terminologie

Voor degene die dit leest en geen technische achtergrond heeft, hierbij een uitleg van veel gebruikte technische termen binnen mijn verslag.

HTML: HTML (Hyper Text Markup Language) is een opmaaktaal waarmee het mogelijk is om documenten te schrijven. HTML is de standaard opmaaktaal voor webpagina's en bestaat al sinds 1997.2

Twig: Twig is een templating engine voor PHP. Twig code wordt gecompileerd naar PHP maar is een stuk beter leesbaar. Ook maakt Twig bepaalde PHP functionaliteit een stuk makkelijker om te gebruiken.3

CSS: CSS (Cascading Style Sheets) zijn een mogelijkheid om webpagina's mee op te maken. In CSS bestanden verwijs je naar HTML elementen om deze vervolgens te stylen.4

Browser Prefixes: Sommige CSS heeft browser prefixes nodig om te kunnen functioneren in verschillende browsers. Hieronder volgt een code voorbeeld:

-webkit-transition: all 100ms ease; // Chrome, Safari, newer versions of Opera.
-moz-transition: all 100ms ease; // Firefox
-ms-transition: all 100ms ease; // Internet Explorer
-o-transition: all 100ms ease; // Opera
transition: all 100ms ease;

JavaScript: Een veelgebruikte scripttaal waarmee het mogelijk is om de DOM (het HTML document) te manipuleren.5

Parameter: een vooraf ingesteld waarde die je meegeeft aan een JavaScript functie. Deze functie kan dan gebruik maken van deze waarde. Hier volgt een voorbeeld:

var myName = "David";

function sayName (name) {
  console.log("Mijn naam is " + name);
};

sayName(myName); // Toont 'Mijn naam is David' in de console

Library: een bibliotheek van vooraf geschreven code. Deze code maakt ontwikkeling van applicaties makkelijker. Als ik het tijdens dit verslag heb over een library praat ik over het algemeen over een JavaScript library.

Component: Een op zichzelf staand stuk code. Een component moet op zichzelf staand kunnen functioneren en geen afhankelijkheid hebben van andere componenten. In mijn project bestaat een component over het algemeen uit twee of drie dingen:

  • Een Twig bestand waarin alle HTML/Twig logica zich bevindt.
  • Een SASS bestand waarin de styling zich bevindt.
  • Optioneel: een JavaScript bestand waarin de functionaliteit van het component zich bevindt.

Refactoring: Het aanpassen van bepaalde code naar een nieuwe (verbeterde) versie. Dit wordt vaak gedaan om de leesbaarheid en de performance van code te verbeteren.

Content Management Sytem: Een Content Management System (ook wel CMS genoemd) bevindt zich over het algemeen aan de achterkant van een website. Via een CMS is het mogelijk om content in te voeren in pagina's en in sommige gevallen kan je er zelfs pagina's mee aanmaken.6

Automated Task Runner: Met een task runner is het mogelijk om je ontwikkelingsproces te stroomlijnen. Denk hierbij aan het automatisch compileren van code wanneer je iets aanpast en het up to date houden van verschillende browsers.

Module Bundler: In mijn project gebruik ik een module bundler om mijn JavaScript bestanden samen te voegen. Het is handig om functionaliteit om te delen in kleinere, beter onderhoudbare bestanden.

Minification: Minifyen is het verwijderen van alle onnodige karakters van stukken code zonder de functionaliteit te veranderen. Dit wordt gedaan om de bestandsgrote te verkleinen en de uitvoeringssnelheid te vergroten.

Slicen: Het nabouwen van een Photoshop ontwerp met behulp van HTML en CSS. Een gesliced ontwerp bevat nog geen functionaliteit.

1.4Probleem analyse

Momkai heeft jaren geleden haar eigen CMS ontwikkeld. Dit CMS wordt ook wel Mimoto genoemd. Mimoto is gebouwd met behulp van Adobe Flash. Adobe Flash wordt door steeds minder browsers ondersteunt en is eigenlijk compleet verouderd.7

Ook is Mimoto niet responsive en mist het veel functionaliteit wat een modern CMS wel biedt. Kortom, Mimoto voldoet dus niet meer aan de huidige eisen van Momkai en haar klanten.

Een paar screenshots van het oude Mimoto.

Mimoto wordt door Momkai gebruikt bij interne en externe projecten. Echter, Mimoto is momenteel zo verouderd dat Momkai bij een nieuw project niet eens meer het voorstel aan de klant doet om Mimoto te gaan gebruiken.

Bij alle nieuwe projecten van Momkai wordt de back-end dus uitbesteedt. Dit heeft voor Momkai meerdere nadelen, zijnde:

  • Mislopen van back-end werkzaamheden.
  • Minder controle over de front-end code aangezien deze door de back-end partij geïmplementeerd dient te worden.
  • Samenwerking met een andere partij kost meer tijd en moeite.

Stakeholders

Het is dus hoog tijd om Mimoto eens grondig op de schop te nemen. Er zijn meerdere stakeholders die baat hebben bij succes van dit project.

We beginnen met de klant. Momkai staat bekend om haar uitstekende kwaliteit en oog voor detail. Wanneer de back-end uitbesteedt wordt kan Momkai hier geen support bij bieden. Als klanten de mogelijkheid hebben om Mimoto af te nemen kan Momkai deze klanten de kwaliteit leveren die zij verdienen.

Ook de werknemers van Momkai zijn gebaat bij een verbeterde versie van Mimoto. Veel interne software maakt namelijk gebruik van het oude Mimoto, wat geen pretje is om te onderhouden of te gebruiken. De nieuwe versie van Mimoto wordt een stuk gebruiksvriendelijker voor developers. Moderne technieken, een goede code structuur en correcte naamgeving zijn belangrijk voor dit project.

En de laatste belanghebbende, Momkai zelf. Met de nieuwe versie van Mimoto kan Momkai nog meer kwaliteit en service bieden dan voorheen. Ook levert het meer werk op, wat weer leidt tot meer inkomsten.

Flexibiliteit

Momkai en ikzelf hebben al veel ervaring op het gebied van applicatieontwikkeling. Een vaak terugkerend probleem met een grote applicaties zoals een CMS is dat het moeilijk is om functionaliteit toe te voegen of te hergebruiken voor andere doeleinden.

Elk project heeft namelijk zijn specifieke eisen en het zou veel ontwikkeltijd schelen als Mimoto gaat werken als een library. Op deze manier wordt onderhoudt minder moeilijk en kan er gemakkelijk op Mimoto door ontwikkeld worden. Ook heb je dan zelf de keuze welke delen van de library je wilt gaan gebruiken.

Design challenge

Bij dit project komt functie voor vorm. Ik ga mij tijdens het traject dan ook niet te veel bezighouden met hoe dingen eruitzien maar meer met hoe ze werken. Het was dan ook belangrijk dat de techniek sterk naar voren kwam in mijn design challenge.

Als design challenge heb ik dan ook gekozen:

Hoe kan ik het CMS van Momkai (Mimoto) updaten en tegelijkertijd zo opzetten dat het qua techniek herbruikbaar en bij de tijd is?

1.5Probleemoplossing

Als oplossing voor het probleem wat ik hierboven beschreven heb wil ik Mimoto opnieuw gaan bouwen. Hierbij wil ik gebruik maken van hedendaagse technieken zoals HTML5, CSS3 en JavaScript.

Tijdens dit project zal ik mij voornamelijk richting op de front-end en alles wat daarmee te maken heeft aangezien daar mijn expertise ligt. De back-end (en grotendeels het concept) zal ontwikkeld worden door Sebastian Kersten. Ik zal tijdens dit project dan ook intensief met hem samenwerken en overleggen.

Modulair

Zoals al eerder uitgelegd is het de bedoeling Mimoto op te zetten als een library. Op deze manier kan het in de toekomst met relatief kleine aanpassingen ook voor andere doeleinden ingezet worden. Om dit te bereiken dient Mimoto opgebouwd te worden uit componenten die elk hun eigen stuk functionaliteit bevatten.

Met deze componenten is het dan mogelijk om ook andere interfaces te bouwen. Het is bijvoorbeeld de bedoeling dat ook Respondens (het CMS van De Correspondent) opnieuw gebouwd wordt met behulp van componenten uit Mimoto.

Omdat een modulaire opzet het belangrijkste aspect is van dit project zal ik mij tijdens de ontwikkeling voornamelijk richting op de belangrijke componenten. In het geval van een CMS zijn dit formulier elementen. Een CMS kan namelijk op 10.000 verschillende manieren ontworpen worden, één ding hebben ze altijd gemeen: er wordt een hoop gebruik gemaakt van formulieren en alles wat hierbij hoort.

Het is dan ook van groot belang dat deze componenten zo opgezet worden dat ze simpel hergebruikt kunnen worden voor andere doeleinden. Om dit te bereiken is het belangrijk om al de functionaliteit geïsoleerd op te zetten. Hoe ik dit precies ga doen ga ik later nog verder op in.

Ontwerp

Het visual design wordt gedaan door ontwerpers die werkzaam zijn bij Momkai en De Correspondent. Ik zal wel bij dit proces betrokken zijn maar niet het daadwerkelijke ontwerp maken.

In de fases die we tijdens dit project doorlopen is ontwerp nog geen belangrijke factor. De techniek en functionaliteit van Mimoto is voor nu nog een stuk belangrijker dan hoe het eruit ziet.

1.6List of requirements

Om een project succes te laten verlopen is het belangrijk om je prioriteiten op orde te hebben. Om dit te bereiken heb ik met behulp van de MoSCoW-methode een lijst opgesteld van prioriteiten.8

Must have

  • Een responsive interface
  • Front-end code die modulair opgezet is
  • Ontwikkeld met behulp van HTML5, CSS3 en JavaScript
  • Ondersteuning van meerdere formulier elementen (foto upload, tekst input, checkboxes, radio buttons en dropdowns)
  • Gebruiksvriendelijke interface, gevalideerd door middel van expert reviews en/of usability tests
  • Validatie bij gebruik formulier element én bij versturen formulier

Should have

  • Login
  • Notificaties

Could have

  • Ondersteuning voor meerdere gebruikers inclusief verschillende rechten
  • Mogelijkheid om te communiceren met andere gebruikers
  • Live preview bij aanpassen content
  • Multi-factor authentication
  • Video upload (inclusief preview)
  • Theming
  • Mogelijkheid om met verschillende gebruikers op dezelfde pagina te werken
  • Grafieken en statistieken

Won't have

  • Ondersteuning voor vertalingen

Veel van de punten die nog niet in de eerste versie van Mimoto komen vind je wel terug in de meeste andere varianten van een open-source CMS. Het is dan ook de bedoeling dat deze functionaliteiten in de toekomst wel toegevoegd worden maar daar is simpelweg nu nog geen tijd voor. Twintig weken is een korte periode om iets te realiseren dus we hebben besloten om voor nu alleen nog de basis functionaliteit te ontwikkelen.

Echter, de afloop van mijn afstudeerproject betekent dus niet het einde van het Mimoto project. Na mijn afstudeerproject ga ik nog een lange tijd samen met Sebastian (en waarschijnlijk ook andere Momkai collega's) verder met de ontwikkeling van Mimoto. We willen van Mimoto graag een complete toolkit maken die door veel mensen met gemak en plezier gebruikt wordt.

2.1Insteek

Op voorhand wist ik dat dit een uitdagend project zou worden. Ik had veel vrijheid waar ik gebruik van kon maken, wat dingen er soms niet makkelijker op maakte.

Het was aan het begin namelijk nog niet helemaal duidelijk wat er allemaal gebouwd diende te worden. We wisten dat we formulier elementen nodig zouden hebben. Ook hadden we natuurlijk een interface nodig om deze elementen in te kunnen gebruiken. Wat we nog niet wisten waren de specifieke componenten die we hier voor nodig zouden hebben.

In het verleden is er ooit een formulier pagina ontworpen voor Respondens. Deze pagina is echter nooit ontwikkeld voor de huidige versie van Respondens. Wel bevatte deze pagina een hoop formulier elementen die we ook nodig gingen hebben voor Mimoto. We hebben toen besloten om dit design aan te houden als richtlijn voor de eerste opzet van de formulier elementen.

De huidige versie van Respondens heeft een goed ontworpen en gebruiksvriendelijke interface. Echter is deze interface niet responsive en kan de code ook wel een opfrisbeurt gebruiken. Om beide problemen te tackelen hebben we besloten om de Respondens interface opnieuw te bouwen. Uiteraard werd het deze keer wel responsive opgezet en ook beter gestructureerd qua code. Deze interface konden we dan gaan gebruiken voor Mimoto.

Het is belangrijk om te begrijpen dat dit project anders was dan alle projecten waar ik ooit eerder aan gewerkt heb. Het ontwerpproces was zo flexibel en variërend dat het soms moeilijk was om hier structuur in aan te brengen. De ene week werkten we aan een functionaliteit om er de volgende week weer achter te komen dat dit misschien toch maar anders moest.

Omdat het dus nog niet compleet duidelijk was wat er allemaal gebouwd diende te worden besloten Sebastian en ik te werken via scrum. Elke sprint duurde één week en via Trello hielden we bij waaraan gewerkt werd en wat er nog gedaan moest worden. Op deze manier was het relatief makkelijk om het overzicht te bewaren over de verschillende werkzaamheden en prioriteiten.

Normaliter verloopt een project op één van de volgende manieren:

  • Er wordt een waterval principe gehanteerd, design levert een ontwerp op wat door front-end gebouwd wordt. Hierna implementeert weer iemand anders de back-end en je hebt een werkend product.
  • Er wordt gewerkt via scrum of een andere agile methode. Dit zorgt ervoor dat er verschillende disciplines tegelijk aan hetzelfde component werken. Over het algemeen heb je bij scrum complete teams (projectmanager, designer, front-end developer, interaction designer) tot je beschikking. Meestal wordt er gewerkt met sprints en werken de teams tot een gezamenlijk doel.
  • Alles wordt door één persoon gedaan (niet aan te raden). In sommige gevallen leidt dit tot een chaotisch ontwerpproces. Wel kan deze manier enorm leuk zijn omdat het veel verantwoordelijkheid met zich meebrengt.

Ons project was eigenlijk een combinatie van de laatste twee opties. Sebastian en ik werkten veel langs elkaar heen omdat Sebastian zich richtte op de back-end en ik mij op de front-end. Dit zorgde ervoor dat we veel dingen zelf moesten uitzoeken.

Wel werd er veel overlegd tussen ons om te bepalen wat er gedaan moest worden en waar we ons op dat moment bevonden. We hadden bijvoorbeeld elke week een planning bijeenkomst waarin wij begeleid door René Clerc (projectleider bij De Correspondent). In deze meetings werden onze Trello lijstjes weer bijgewerkt tot de laatste versies.

Uiteraard heb ik ook mijn collega's bij Momkai veel betrokken bij dit project. Als ik hulp nodig had bij een ontwerp kwestie kon ik altijd terecht bij één van mijn visual design collega's. Ik heb hen dan ook een hoop vragen gesteld, variërend van vragen over uitlijning op bepaalde resoluties tot het verzoek om nieuwe iconen te ontwerpen.

Ook heb ik geprobeerd om zo vaak mogelijk te sparren met een interactie designer over bepaalde functionaliteiten. Op deze manier kwam ik vaak tot inzichten waar ik zonder hulp niet snel tot gekomen zou zijn. Dit heeft ertoe geleidt dat de formulier elementen qua functionaliteit goed in elkaar zitten.

2.2Development tools

Als front-end developer maak je tijdens het ontwikkelingsproces veel gebruik van verschillende tools. Hieronder laat ik zien wat mijn meest gebruikte tools zijn en waar ik deze voor inzette.

Github

Github is een website die met behulp van Git het mogelijk maakt om in groepen aan dezelfde software te werken1. Github heeft concurrentie van andere diensten (zoals Bitbucket) die ongeveer hetzelfde aanbieden maar is verreweg de grootste speler.

Ik gebruik Github dan ook al sinds ik ben begonnen als front-end developer en ik zie geen reden om hier de komende tijd nog van af te stappen. Github heeft een gigantische community achter zich en alle functionaliteit die je als developer nodig hebt. Als je geïnteresseerd bent in hoe Github precies werkt kan je er hier meer over lezen.

Voor dit project gebruikten Sebastian en ik Github uiteraard veel. Al onze code werd via Github beheerd en opgeslagen. Op deze manier konden fouten makkelijk teruggedraaid worden en konden Sebastian en ik goed naast elkaar werken. Ook was onze code veilig mocht één van onze computers onverwachts stuk gaan.

SourceTree

SourceTree is een programma waarmee het mogelijk is om te communiceren met diensten die Git gebruiken (zoals Github). Het is mogelijk om met Git te werken via de command line alleen vergt dit de beheersing van veel commands.

Voor mensen die er geen zin in hebben om deze commands te leren zijn er programma's zoals SourceTree. Sourcetree maakt het mogelijk om via een interface Git commands uit te voeren. Dit zorgt ervoor dat je nooit in de buurt van een command line hoeft te komen.

Node en NPM

Node zorgt ervoor dat het mogelijk is om JavaScript uit te voeren via je console. Op deze manier is het mogelijk om tools zoals Gulp en Grunt te gebruiken, hierover later meer.

Als je Node installeert krijg je gelijk de Node Package Manager (NPM) meegeleverd. NPM is de meest gebruikte tool om packages te installeren en beheren. Op deze website kun je meer lezen over NPM en al haar packages.

Je kunt via NPM op een hele makkelijke manier een package installeren en opslaan in een zogenaamd package.json bestand. Als iemand anders dan je project binnen haalt kan hij via het volgende commando: npm install alle dependencies van dat project installeren.

Ik heb Node voor dit project gebruikt om Gulp te draaien. Ook heb ik via NPM alle packages die ik bij dit project gebruik opgeslagen en geïnstalleerd.

Gulp

Gulp is een automated task runner. Je kunt voor Gulp tasks schrijven die je leven tijdens de ontwikkeling van een project een stuk aangenamer maken.

Gulp heeft concurrentie van andere task runners, waaronder Grunt. Er zijn meerdere redenen waarom ik gekozen heb voor Gulp bij dit project (en bij vele andere projecten). Een goede vergelijking van Gulp vs Grunt kan hier gelezen worden.

Ik gebruikte Gulp voor een hoop tasks tijdens mijn project. Een belangrijke was bijvoorbeeld de SASS task, die ervoor zorgde dat mijn SASS elke keer opnieuw gecompileerd werd naar CSS als ik een aanpassing deed. Ook zorgde deze task ervoor dat de gecompileerde CSS gelijk geminified werd en de nodige browser prefixes werden toegevoegd.

Een andere veel gebruikte task was de zogenaamde Browsersync task. Wanneer Browsersync geactiveerd wordt zorgt dit ervoor dat je website lokaal bereikbaar is vanaf het IP-adres 127.0.0.1. Als je dan aanpassingen maakt in je JavaScript of CSS zorgt Browsersync ervoor dat deze aanpassingen gelijk gestreamd worden naar de browser. Op deze manier heb je dus je laatste code zonder steeds zelf een page refresh te moeten uitvoeren.

Webpack

Webpack is een module bundler. Met Webpack kun je ontzettend veel dingen doen maar ik gebruik het voornamelijk om JavaScript te compileren.

Webpack is hier erg goed in en het is zelfs mogelijk om voor verschillende ontwikkelingsomgevingen verschillende instellingen te hebben. Denk hierbij aan bijvoorbeeld andere instellingen voor development dan voor live.

Eén van de belangrijkste redenen om met een module bundler te werken is zodat je je JavaScript mooi kunt opdelen in hapklare brokken. Webpack is een veelgebruikte keuze omdat Webpack de twee meest gebruikte varianten van JavaScript modules ondersteunt: CommonJS en AMD.

Andere module bundlers zoals Browserify en RequireJS verwachten modules in één van de twee formaten, Webpack kan met allebei overweg.

Voor dit project heb ik ervoor gekozen om mijn JavaScript modules te schrijven via het CommonJS formaat. Waarom ik dit heb gedaan licht ik later nog toe.

Overige tools

Buiten al deze tools heb ik natuurlijk ook nog wat programma's gebruikt tijdens mijn proces. Om mijn code te schrijven heb ik het programma PhpStorm gebruikt. PhpStorm biedt ontzettend veel functionaliteit en is dan ook een fijn programma om mee te werken.

Ook werkte ik met een lokale database. Om aanpassingen in deze database te kunnen doen gebruikte ik het programma Sequel Pro. Sequel pro maakt het makkelijker om te werken met databases.

Om de designs die gemaakt werden te bekijken gebruikte ik uiteraard het programma Adobe Photoshop. Ik ben geen grote fan van Photoshop maar kan er goed genoeg mee overweg om er geen last van te hebben.

JavaScript module setup

Bij vrijwel alle Momkai projecten wordt er gebruik gemaakt van de open-source Conditioner.js library. Met deze library is het mogelijk om JavaScript modules in te laden wanneer er aan bepaalde condities voldaan wordt.

In de praktijk ziet dit er als volgt uit:

<a href="http://maps.google.com/?ll=51.741,3.822" data-module="ui/Map" data-conditions="media:{(min-width:40em)} and element:{was visible}"> ... </a>

Als je geen condities instelt (zoals wij over het algemeen doen bij Momkai) wordt de JavaScript gewoon geladen wanneer het component waar de JavaScript bij hoort gebruikt wordt. Dit is handig zodat je geen if-statements hoeft te schrijven in je initiatie JavaScript bestand.

Het was dus ook de bedoeling om Conditioner te gaan gebruiken bij dit project. Echter, toen ik bezig was met de opzet van mijn development setup kreeg ik Conditioner niet werkend. Na wat onderzoek gedaan te hebben kwam ik tot de conclusie dat wij een custom build van Conditioner gebruikten zodat het mogelijk was om JavaScript modules te schrijven in het CommonJS formaat. Oorspronkelijk ondersteunt Conditioner alleen AMD JavaScript modules.

Normaliter schreef ik mijn JavaScript modules altijd via het CommonJS formaat. Aangezien ik Conditioner graag wou gebruiken besloot ik om voor dit project mijn JavaScript modules via het AMD formaat te schrijven.

Echter kreeg ik Conditioner alsnog niet werkend in combinatie met Webpack. Conditioner heeft om te kunnen werken een loader zoals bijvoorbeeld RequireJS nodig. Echter kan Webpack alles zelf al wat RequireJS ook kan dus is het nogal raar om verschillende dingen te gebruiken die hetzelfde doen.

Om mijn probleem op te lossen besloot ik om via Github contact te zoeken met de ontwikkelaar van Conditioner. Daar stelde ik hem de volgende vraag.

Na een paar berichten heen en weer kreeg ik eindelijk een werkend voorbeeld van hoe het mogelijk was om Conditioner te gebruiken in combinatie met Webpack. Dit voorbeeld was echter zo ingewikkeld en achterhaald dat ik besloten heb om Conditioner niet te gebruiken binnen Mimoto. Voor deze beslissing had ik de volgende argumenten:

  • Conditioner heeft een extra loader nodig om te functioneren terwijl Webpack al alles kan wat die loaders ook kunnen. Het gebruik van een extra loader is dan ook overkill.
  • De performance van Conditioner is minder goed dan de 'oldskool' manier van if-statements.

Omdat ik dus geen Conditioner kon gebruiken binnen dit project moest ik mijn JavaScript modules op de ouderwetse manier inladen. Ik zal nu uitleggen hoe dat precies te werk is gegaan.

JavaScript modules worden over het algemeen geschreven als JavaScript objecten met een constructor functie. Dit is een JavaScript object:

var myObject = {
  // Hier kan je dingen doen
};

Op deze manier is het niet mogelijk om nieuwe instanties van dit object aan te maken. Dit komt omdat dit een object literal is. Wat dat precies is leg ik later nog uit. Om hier een object constructor van te maken dienen er wat aanpassingen gedaan te worden.

var Human = function (name) {
  this.name = name;
};

Human.prototype = {
  introduce: function () {
    console.log("Mijn naam is " + this.name);
  }
};

var human = new Human("David");
human.introduce(); // Toont 'Mijn naam is David' in de console.

Hierboven definieer ik een variabele genaamd Human. Aan deze variabele zit een functie gekoppeld. Deze functie wordt gebruikt als een object en er wordt via het prototype een functie genaamd introduce toegevoegd.

Met de JavaScript operator new maak ik dan een nieuwe instantie aan van het Human object. Deze instantie sla ik op in de variabele human. Via deze variabele kan ik daarna de functies aanroepen die horen bij het Human object.

Vrijwel alle componenten die ik heb ontwikkeld bevatten een JavaScript module. Omdat ik mijn modules beheer met Webpack is de syntax iets anders. Webpack werkt namelijk met de module.exports syntax. Mijn JavaScript modules zagen er daardoor zo uit:

module.exports = function () {
  this.init();
};

module.exports.prototype = {
  init: function () {
    // Begin hier met het uitvoeren van andere code
  }
};

Als ik dan een module wou gebruiken voegde ik deze toe aan mijn mimoto.cms.js bestand. In dit bestand wordt alle JavaScript van Mimoto geïnitialiseerd. Webpack zorgt er uiteindelijk voor dat alle modules ook daadwerkelijk worden samengevoegd en geminified.

Het toevoegen van een JavaScript module aan mijn initiatie bestand is extreem simpel om te doen. Het kan namelijk al met de volgende code:

var headerView = require('./views/Header');

Nadat een module is toegevoegd moet hij geïnitialiseerd worden wil je hem kunnen gebruiken. Dat kan met de eerder getoonde new operator. Hieronder volgt een voorbeeld vanuit mimoto.cms.js waarin modules worden ingeladen en aangeroepen.

var headerView = require('./views/Header'); // Require de module en sla hem op in een variabele

document.addEventListener('DOMContentLoaded', function () { // JavaScript binnen dit blok wordt pas uitgevoerd wanneer de pagina geladen is

  var navigation = document.querySelector('.js-navigation'); // Variabelen waarin HTML elementen worden opgeslagen
  var header = document.querySelector('.js-header');

  if (navigation && header) { // Check of de HTML elementen die nodig zijn voor de HeaderView module bestaan
    new HeaderView(header); // Initieer HeaderView module
  }

}, false);

Vaak geef ik bij mijn modules het element waar hij bij hoort mee als parameter. Je ziet in het voorbeeld hierboven dat ik de variabele header meegeef aan de HeaderView instantie. Op deze manier weet de HeaderView instantie meteen welk element hij dient te besturen.

4.2Werking FormValidation

Wil je meer over JavaScript data types en literals te weten komen lees dan eens het volgende artikel. Omdat het nogal een uitgebreid artikel is zal ik zelf ook een kort voorbeeld geven van hoe een object literal precies werkt.

var human = {

Hierboven wordt een JavaScript object gedefinieerd en toegekend aan de variabele human. Dit object bevat twee functies, speak en run. Deze functies kunnen op de volgende manier worden aangeroepen: human.speak(); of human.run();.

Data-attributen

Als je gebruik wilt maken van FV dien je als eerste data-attributen toe te voegen aan je formulier element. De volgende data-attributen worden ondersteunt voor formulier elementen waarin getypt kan worden:

  • Min-length: de minimale lengte van de input waarde.
  • Max-length: de minimale lengte van de input waarde.
  • No-numbers: blokkeert het gebruik van nummers in de input waarde.
  • Min-numbers: minimaal aantal nummers wat de input waarde moet bevatten.
  • Max-numbers: maximaal aantal nummers wat de input waarde mag bevatten.
  • No-special-characters: blokkeert het gebruik van speciale karakters in de input waarde.
  • Min-special-characters: minimaal aantal speciale karakters wat de input waarde moet bevatten.
  • Max-special-characters: maximaal aantal speciale karakters wat de input waarde mag bevatten.
  • Custom-regex: een custom reguliere expressie waartegen de input waarde gevalideerd wordt.
  • Error-message: moet gebruikt worden in combinatie met het custom-regex data-attribuut. Via dit data-attribuut kan een custom error message ingesteld worden welke getoond wordt wanneer de validatie met de custom-regex faalt.

Ook zijn er nog twee data-attributen die gebruikt kunnen worden in combinatie met checkboxes:

  • Min-checked: het minimale aantal checkboxes wat geselecteerd dient te worden.
  • Max-checked: het maximale aantal checkboxes wat geselecteerd mag worden.

Buiten al deze data-attributen zijn er ook nog twee data-attributen die gebruikt kunnen worden met radio buttons en dropdowns. Deze formulier elementen zijn per definitie niet verplicht. Door middel van het toevoegen van een data-attribuut kunnen deze elementen verplicht gemaakt worden. Het data-attribuut radio-button-required kan gebruikt worden voor radio buttons en het data-attribuut dropdown-required kan gebruikt worden voor dropdowns.

Ik heb de data-attributen geprefixed met fv zodat het duidelijk is waar de data-attributen voor gebruikt worden. In de praktijk ziet een data-attribuut er dus zo uit: data-fv-min-length="10".

Uitleg functies

Nadat je al je data-attributen naar keuze hebt toegevoegd aan je input element kun je dit element valideren door middel van de volgende functie: FV.validateInput();. Als parameter dien je een element mee te geven wat de parent is van het input element waar je de data-attributen aan hebt toegevoegd. Je HTML dient er dus ongeveer als volgt uit te zien:

<div class="textline-parent js-textline-parent"> // De parent van het input element 
  <input class="textline js-textline" type="text" placeholder="Uw naam" data-fv-min-length="5" data-fv-max-length="15" data-fv-no-numbers data-fv-no-special-characters />
</div>

Als je dan dit input element wilt valideren kan dat met de volgende JavaScript:

var textlineParent = document.querySelector('.js-textline-parent');

FV.validateInput(textlineParent);

De validateInput functie doet niks anders dan het valideren van de waarde uit het input element en geeft het resultaat hiervan terug. Als de waarde door de validatie heen komt wordt er teruggegeven dat het valideren gelukt is. Komt de waarde niet door de validatie heen wordt er ook nog een error message teruggegeven.

Dit is hoe de validateInput functie er exact uitziet:

validateInput: function (element) {

  this.setVariables(element);
  this.setInputOptions();
  this.setResult(true);
  this.handleValidation();

  return this.result;

}

Zoals je ziet verwacht validateInput een element als parameter en geeft hij het resultaat terug wanneer hij klaar is met het uitvoeren van de vier andere functies die hij aanroept.

De eerste functie die validateInput aanroept is setVariables. Voordat ik verder inga op de werking van de setVariables functie zal ik eerst wat toelichting op JavaScript variabelen geven. Een normale JavaScript variabele wordt gedefinieerd op de volgende manier:

var variable = "Hello world!";
console.log(variable); // Toont 'Hello world!' in de console

Als je op deze manier een variabele definieert in een functie is hij buiten deze functie niet toegankelijk. Het volgende voorbeeld werkt dus niet:

// Definieer twee functies
function setVariable ()  {
  var variable = "Hello world!";
};

function logVariable () {
  console.log(variable);
};

// Voer de functies uit
setVariable();
logVariable(); // Toont de volgende error in de console: 'Uncaught ReferenceError: variable is not defined'

Je kunt variabelen wel bereikbaar maken in verschillende functies binnen een object literal. Het bovenstaande voorbeeld kan dus met relatief kleine aanpassingen wel werkend gemaakt worden:

// Definieer een object literal welke de twee functies bevat
var variableObject = {

  setVariable: function ()  {
    this.variable = "Hello world!";
  },

  logVariable: function () {
    console.log(this.variable);
  };

};

// Voer de functies uit
variableObject.setVariable();
variableObject.logVariable(); // Toont 'Hello world!' in de console

Buiten de aanpassingen naar de object literal notatie is de grootste aanpassing het this keyword. Alle variabelen die in een object literal via het this keyword worden gedefinieerd zijn binnen elke andere functie binnen hetzelfde object te benaderen.

Nu je enigszins begrijpt hoe variabelen werken binnen JavaScript kan ik uitleggen hoe de setVariables functie werkt. Dit is hoe hij eruit ziet:

setVariables: function (element) {

  this.el = element;

  this.textline = this.el.querySelector('.js-textline');
  this.checkboxes = this.el.querySelectorAll('.js-checkbox');
  this.radioButtons = this.el.querySelectorAll('.js-radio-button');

  if (this.textline) {
    this.input = this.textline;
    this.value = this.input.value;
  } else if (this.checkboxes.length) {
    this.input = this.checkboxes[0];
    this.countChecked();
  } else if (this.radioButtons.length) {
    this.input = this.radioButtons[0];
  }

}

Het eerste wat hier gebeurd is dat het element wat setVariables meekrijgt als parameter wordt gedefinieerd als this.el. Daarna wordt er binnen this.el gezocht naar verschillende classes. De HTML elementen die deze classes bevatten worden opgeslagen in variabelen. Dat wordt gedaan met de volgende code:

this.textline = this.el.querySelector('.js-textline');
this.checkboxes = this.el.querySelectorAll('.js-checkbox');
this.radioButtons = this.el.querySelectorAll('.js-radio-button');

Nadat deze variabelen zijn gedefinieerd wordt er door middel van een if-statement gekeken of de daadwerkelijke variabelen ook echt gevuld zijn. Er worden verschillende dingen gedaan wanneer er verschillende variabelen gevuld zijn. Voor dit voorbeeld gaan we nu er nu even vanuit dat this.textline gevuld is. Dat betekent dat de volgende code wordt uitgevoerd:

this.input = this.textline;
this.value = this.input.value;

Na het uitvoeren van de setVariables functie wordt de setInputOptions functie uitgevoerd. Deze functie ziet er als volgt uit:

setInputOptions: function () {

  this.minLength = this.input.getAttribute('data-fv-min-length');
  this.maxLength = this.input.getAttribute('data-fv-max-length');

  this.noNumbers = this.input.hasAttribute('data-fv-no-numbers');
  this.minNumbers = this.input.getAttribute('data-fv-min-numbers');
  this.maxNumbers = this.input.getAttribute('data-fv-max-numbers');

  this.noSpecialCharacters = this.input.hasAttribute('data-fv-no-special-characters');
  this.minSpecialCharacters = this.input.getAttribute('data-fv-min-special-characters');
  this.maxSpecialCharacters = this.input.getAttribute('data-fv-max-special-characters');

  this.minChecked = this.input.getAttribute('data-fv-min-checked');
  this.maxChecked = this.input.getAttribute('data-fv-max-checked');

  this.ifChecked = this.input.type == 'radio';

  this.customRegex = this.input.getAttribute('data-fv-regex');
  this.errorMessage = this.input.getAttribute('data-fv-error-message');

  this.setValidationOptions();

}

De bovenstaande functie leest de waardes uit van alle data-attributen van this.input. Bevat this.input een data-attribuut niet dan wordt de waarde ingesteld als een lege string, oftewel ''. Nadat alle waardes zijn uitgelezen en opgeslagen wordt de setValidationOptions functie uitgevoerd. Die ziet er zo uit:

setValidationOptions: function () {

  this.validateMinLength = this.minLength ? true : false;
  this.validateMaxLength = this.maxLength ? true : false;
  this.validateNoNumbers = this.noNumbers ? true : false;
  this.validateMinNumbers = this.minNumbers ? true : false;
  this.validateMaxNumbers = this.maxNumbers ? true : false;
  this.validateNoSpecialCharacters = this.noSpecialCharacters ? true : false;
  this.validateMinSpecialCharacters = this.minSpecialCharacters ? true : false;
  this.validateMaxSpecialCharacters = this.maxSpecialCharacters ? true : false;
  this.validateMinChecked = this.minChecked ? true : false;
  this.validateMaxChecked = this.maxChecked ? true : false;
  this.validateIfChecked = this.ifChecked ? true : false;
  this.validateCustomRegex = this.customRegex ? true : false;

}

Deze functie kijkt of de variabelen die in de setInputOptions functie zijn ingesteld ook echt waardes bevatten. Bevatten ze een waarde dan wordt de bijbehorende variabele op true ingesteld. Zo niet, dan wordt de bijhorende variabele op false ingesteld (een lege string is in JavaScript geen waarde). Kortom, deze functie stelt in waar er allemaal op gevalideerd dient te worden.

Hierna wordt een belangrijke functie binnen FV uitgevoerd, namelijk de setResult functie. De setResult functie ziet er als volgt uit:

setResult: function (passed, message) {

  this.result = {};

  this.result.passed = passed;

  if (message) {
    this.result.message = message;
  }

}

Via deze functie is het mogelijk om het resultaat in te stellen wat uit eindelijk door de validateInput functie wordt teruggegeven. Het is mogelijk om twee parameters mee te geven aan setResult. De eerste parameter is een boolean (kan alleen true of false zijn) en gaat over of de validatie is gelukt. De tweede parameter is een optioneel bericht. Deze parameter wordt alleen gebruikt wanneer de validatie niet gelukt is.

In de validateInput functie wordt het resultaat ingesteld op true aangezien de validatie initieel altijd gelukt is. Via de handleValidation functie wordt dan de daadwerkelijke validatie gedaan en wordt zo nodig het resultaat geüpdatet. Dit is hoe de handleValidation functie eruitziet:

handleValidation: function () {

  if (this.validateMinLength) this.checkMinLength();

  if (this.validateMaxLength) this.checkMaxLength();

  if (this.validateNoNumbers) this.checkNoNumbers();

  if (this.validateMinNumbers) this.checkMinNumbers();

  if (this.validateMaxNumbers) this.checkMaxNumbers();

  if (this.validateNoSpecialCharacters) this.checkNoSpecialCharacters();

  if (this.validateMinSpecialCharacters) this.checkMinSpecialCharacters();

  if (this.validateMaxSpecialCharacters) this.checkMaxSpecialCharacters();

  if (this.validateMinChecked) this.checkMinChecked();

  if (this.validateMaxChecked) this.checkMaxChecked();

  if (this.validateIfChecked) this.checkIfChecked();

  if (this.validateCustomRegex) this.checkCustomRegex();

  this.result.passed ? EH.addValidatedState(this.el) : EH.addErrorState(this.el, this.result.message);

}

De handleValidation functie kijkt naar de variabelen die door setValidationOptions zijn ingesteld. Is een variabele true dan wordt de bijhorende validatie functie uitgevoerd. Nadat alle validatie is gedaan wordt er gekeken wat de waarde van this.result.passed is. Is deze waarde true dan betekent het dat de validatie is geslaagd. Is deze waarde false betekent het dat de validatie mislukt is.

Om de uitleg over FV af te ronden zal ik nog uitleg geven over de daadwerkelijke validatie functies. Ik ga niet in detail in op de werking van al deze functies aangezien ze vrijwel allemaal hetzelfde werken. Wel zal ik de checkNoNumbers functie uitleggen om te demonstreren hoe de validatie functies over het algemeen werken. Dit is de checkNoNumbers functie:

checkNoNumbers: function () {

  var regex = new RegExp("\\d");

  if (regex.test(this.value)) {
    this.setResult(false, "No numbers allowed.");
  }

}

Met het volgende stuk code new RegExp() kan ik een nieuwe reguliere expressie aanmaken. Als parameter geef ik dan de daadwerkelijke reguliere expressie mee. De volgende regel code maakt een nieuwe reguliere expressie aan die checkt of een tekst cijfers bevat.

var regex = new RegExp("\\d");

Om een reguliere expressie te gebruiken dien je een waarde te vergelijken met de reguliere expressie. De reguliere expressie geeft dan als resultaat terug of de waarde voldoet aan de gestelde eisen. Dit doe je op de volgende manier:

var text = "Hello World 1"; // Een tekst die een cijfer bevat
var regex = new RegExp("\\d"); // Reguliere expressie die checkt of een waarde cijfers bevat

var result = regex.test(text); // Resultaat van de check, in dit geval 'true' aangezien de waarde een cijfer bevat

console.log(result); // Toont 'true' in de console

De checkNoNumbers functie werkt bijna hetzelfde als het voorbeeld hierboven. Als eerst wordt dus de reguliere expressie gedefinieerd. Daarna wordt this.value getest ten opzichte van de reguliere expressie. Als het resultaat true is betekent het his.value nummers bevat. Aangezien er geen nummers toegestaan zijn binnen this.value wordt de this.setResult functie aangeroepen en wordt het resultaat ingesteld op false. Dit gebeurt allemaal in het volgende stuk code:

var regex = new RegExp("\\d");

if (regex.test(this.value)) { // Geeft 'true' terug als this.value cijfers bevat
  this.setResult(false, "No numbers allowed.");
}

4.3Werking ErrorHandling

De ErrorHandling library is gebouwd om de state van een formulier element te kunnen veranderen. De oplettende lezer heeft al een glimp opgevangen van hoe deze library gebruikt kan worden. Dit is namelijk de laatste regel van de FV.handleValidation functie:

this.result.passed ? EH.addValidatedState(this.el) : EH.addErrorState(this.el, this.result.message);

De EH variabele verwijst naar mijn ErrorHandling library. Hier zie je dus dat er verschillende states worden toegevoegd. Welke state wordt toegevoegd hangt af van de waarde van de this.result.passed variabele.

Instellen opties

De EH library werkt iets anders dan de FV library. Voordat je EH namelijk kunt gebruiken met je eigen custom HTML dien je eerst de init functie aan te roepen. Die functie ziet er als volgt uit:

  this.errorElement = "p";
  this.errorElementClasses = "error-message";
  this.iconSelectorClass = "icon";

  this.validatedIcon = "#ico-validated";
  this.validatedIconClass = "icon-validated";
  this.errorParentClass = "error-parent";
  this.errorClass = "has-error";
  this.errorIcon = "#ico-error";
  this.errorIconClass = "icon-error";

  if (options.errorElement) this.errorElement = options.errorElement;

  if (options.iconSelectorClass) this.iconSelectorClass = options.iconSelectorClass;

  if (options.validatedClass) this.validatedClass = options.validatedClass;

  if (options.validatedIcon) this.validatedIcon = options.validatedIcon;

  if (options.validatedIconClass) this.validatedIconClass = options.validatedIconClass;

  if (options.errorIcon) this.errorIcon = options.errorIcon;

De bovenstaande functie stelt eerst een hoop default opties in. Daarna kijkt hij of de desbetreffende optie is meegegeven aan de options parameter. Zo ja, dan overschrijft hij de eerder ingestelde default waarde. Dit zijn de opties die je kunt instellen:

  • This.errorElement: het HTML element waarin de error wordt getoond.
  • This.errorElementClasses: de classes die aan this.errorElement worden toegevoegd.
  • This.iconSelectorClass: een HTML class om het SVG element mee te selecteren.
  • This.validatedClass: een HTML class die wordt toegevoegd wanneer de validated status wordt toegevoegd.
  • This.validatedIcon: een ID referentie naar het SVG icoon wat wordt toegevoegd wanneer de validated status wordt toegevoegd.
  • This.validatedIconClass: een HTML class die wordt toegevoegd aan this.iconSelectorClass wanneer de validated status wordt toegevoegd.
  • This.errorParentClass: een HTML class waaraan this.errorElement wordt toegevoegd.
  • This.errorClass: een HTML class die wordt toegevoegd wanneer de error status wordt toegevoegd.
  • This.errorIcon: een ID referentie naar het SVG icoon wat wordt toegevoegd wanneer de error status wordt toegevoegd.
  • This.errorIconClass: een HTML class die wordt toegevoegd aan this.iconSelectorClass wanneer de error status wordt toegevoegd.

Belangrijkste functies

Nadat je je opties hebt ingesteld kun je EH gaan gebruiken. Er zijn eigenlijk maar twee functies die je als gebruiker zult aanroepen. Dat zijn de functies addErrorState en addValidatedState. Deze functies updaten het element wat wordt meegegeven als parameter. Hierbij gebruiken ze de opties die je eerder hebt ingesteld.

De addValidatedState functie kun je gebruiken om een validated state toe te voegen aan een element. De addValidatedState functie doet dit in meerdere stappen. Dit is hoe hij eruit ziet:

addValidatedState: function (element) {

  this.setVariables(element);
  this.clearErrorState();

  if (!this.isValidated) {
    this.addElementClass(this.validatedClass);
    this.addIcon(this.validatedIconClass, this.validatedIcon);
  }

}

Het eerste wat addValidatedState doet is een andere functie genaamd setVariables aanroepen. Dat is de volgende functie:

setVariables: function (element) {

  this.el = element;

  this.errorParent = this.el.querySelector('.' + this.errorParentClass);
  this.iconElement = this.el.querySelector('.' + this.iconSelectorClass);
  this.useElement = this.iconElement.getElementsByTagName('use')[0];

  this.hasError = this.el.classList.contains(this.errorClass);
  this.isValidated = this.el.classList.contains(this.validatedClass);

}

In de setVariables functie worden variabelen ingesteld die later weer gebruikt worden in andere functies. De tweede functie die door addValidatedState wordt aangeroepen is de clearErrorState functie:

clearErrorState: function () {

  if (this.hasError) {
    this.removeElementClass(this.errorClass);
    this.removeErrorElement();
    this.removeIcon(this.errorIconClass);
  }

}

De clearErrorState functie verwijderd alle styling die hoort bij de error state. Dit zorgt ervoor dat als je element error styling had hij nu weer gereset is naar zijn oorspronkelijke state. Hoe de functies die door clearErrorState worden aangeroepen precies werken leg ik later uit.

Nadat de error state is verwijderd wordt er gecontroleerd of het element niet al de gevalideerde state bevat. Bevat het element deze state niet worden er twee functies aangeroepen. Dit zijn de addElementClass en addIcon functies. Dit gebeurt allemaal in het volgende stuk code:

if (!this.isValidated) {
  this.addElementClass(this.validatedClass);
  this.addIcon(this.validatedIconClass, this.validatedIcon);
}

De addElementClass functie doet eigenlijk niks anders dan een class toevoegen aan het element. De class dient meegegeven te worden als parameter. Met de removeElementClass functie (gebruikt in de clearErrorState functie) kan je deze class dan weer verwijderen. Deze functies zien er zo uit:

// Deze functie voegt een class toe aan this.el
addElementClass: function (elementClass) {

  this.el.classList.add(elementClass);

}

// Deze functie verwijderd een class van this.el
removeElementClass: function (elementClass) {

  this.el.classList.remove(elementClass);

}

De addIcon functie doet precies wat je ervan verwacht, hij voegt namelijk een icon toe. Verschillende icons hebben verschillende styling dus dient er ook naast het icon ook een class meegegeven te worden als parameter. De icon en class worden dan toegevoegd aan het element.

Met de removeIcon functie (gebruikt in de clearErrorState functie) kan dit icon dan weer verwijderd worden. Als parameter dient er alleen een class mee gegeven te worden. Dit komt omdat als de class verwijderd wordt de styling van het icon ook verwijderd wordt. Zonder styling is het icon niet zichtbaar. Dit zijn de addIcon en removeIcon functies:

// Deze functie voegt een class en icon toe aan this.el
addIcon: function (iconClass, icon) {

  this.iconElement.classList.add(iconClass);
  this.useElement.setAttributeNS('http://www.w3.org/1999/xlink', 'xlink:href', icon);

}

// Deze functie verwijderd de iconClass van this.el
removeIcon: function (iconClass) {

  this.el.querySelector('.' + this.iconSelectorClass).classList.remove(iconClass);

}

Dit zijn alle stappen die worden doorlopen bij het toevoegen van een validated state. Zoals je ziet kan er nogal veel komen kijken bij het aanroepen van één functie.

Natuurlijk wil je ook de mogelijkheid hebben om een error state te kunnen toevoegen. Dit kan gedaan worden met de addErrorState functie. Zo ziet deze functie eruit:

addErrorState: function (element, message) {

  this.setVariables(element);
  this.clearValidatedState();

  if (!this.hasError) {
    this.addErrorElement(message);
    this.addElementClass(this.errorClass);
    this.addIcon(this.errorIconClass, this.errorIcon);
  } else {
    this.updateErrorElement(message);
  }

}

Zoals je ziet worden er aan deze functie twee parameters meegegeven, element en message. De element parameter vervult dezelfde rol als de element parameter bij de addValidatedState functie. De message parameter bevat de error message die je wilt tonen.

Net zoals bij de addValidatedState functie is de eerste functie die wordt aangeroepen de setVariables functie. Daarna wordt de clearValidatedState functie aangeroepen, die ziet er als volgt uit:

clearValidatedState: function () {

  if (this.isValidated) {
    this.removeElementClass(this.validatedClass);
    this.removeIcon(this.validatedIconClass);
  }

}

De clearValidatedState functie is vrijwel hetzelfde als de clearErrorState functie. Er wordt gecontroleerd of het element de validated state bevat. Is dit het geval dan worden de removeElementClass en removeIcon functies aangeroepen om de validated state te verwijderen.

Nadat de variabelen zijn ingesteld door de setVariables functie en de validated state is verwijderd door de clearValidatedState functie kan de error state toegevoegd worden. De error state bestaat uit drie dingen:

  • Een class die wat error styling toevoegt (zoals bijvoorbeeld een rode border om het input veld heen)
  • Een icon plus bijbehorende class (in het geval van Mimoto een rood kruisje)
  • De daadwerkelijke error (getoond onder het input veld)

De addErrorState functie controleert of het element al de error state bevat. Als dit het geval is dient namelijk alleen de error message geüpdatet te worden. Dit kan gedaan worden met de updateErrorElement functie:

updateErrorElement: function (message) {

  var error = this.errorParent.querySelector('.' + this.errorElementClasses[0]);
  error.innerHTML = message;

}

De updateErrorElement functie zoekt naar een error element in his.errorParent. Dit element wordt opgeslagen in de variabele error. De innerHTML van deze variabele wordt daarna ingesteld op de message parameter.

Als het element nog geen error state bevat dient deze toegevoegd te worden. Hier zijn de volgende drie functies voor nodig:

this.addErrorElement(message);
this.addElementClass(this.errorClass);
this.addIcon(this.errorIconClass, this.errorIcon);

Aangezien ik de addElementClass en addIcon functies al heb uitgelegd zal ik alleen nog de addErrorElement functie uitleggen. Die functie ziet er zo uit:

addErrorElement: function (message) {

  var error = document.createElement(this.errorElement);

  for (var i = 0; i < this.errorElementClasses.length; i++) {

    error.classList.add(this.errorElementClasses[i]);

  }

  error.innerHTML = message;
  this.errorParent.appendChild(error);

}

Het eerste wat er in deze functie gebeurd is de creatie van een nieuw HTML element. Dit element wordt opgeslagen in de variabele error. Daarna worden er door middel van een for loop classes toegevoegd aan het error element.

Nadat het error element is gecreëerd en de classes zijn toegevoegd wordt de message parameter ingesteld als innerHTML van het error element. Als laatste wordt het error element dan ook daadwerkelijk toegevoegd aan de pagina. Dit gebeurt op de volgende regel:

this.errorParent.appendChild(error);

Uiteraard kunnen toegevoegde errors ook weer verwijderd worden. Dit kan gedaan worden met de removeErrorElement functie (gebruikt in de clearErrorState functie). Dit is de removeErrorElement functie:

removeErrorElement: function () {

  var error = this.errorParent.querySelector('.' + this.errorElementClasses[0]);

  if (error) {
    this.errorParent.removeChild(error);
  }

}

Deze functie selecteert het error element en slaat deze op in de variabele error. Daarna wordt er gecontroleerd of het error element daadwerkelijk bestaat. Is dit het geval dan wordt het error element verwijderd uit de HTML pagina.
